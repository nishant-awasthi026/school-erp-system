1. What the School ERP Platform Is
The School ERP Platform is a complete, cloud-hosted school management system delivered as a multi-tenant SaaS product. The word 'complete' is intentional — this is not a tool that solves one problem for a school. It is the single software system through which a school runs every administrative and academic process it has, from the moment a student applies for admission to the day they walk out with their final report card.

The platform is built on a principle of radical simplicity for the user and radical sophistication beneath the surface. A teacher should be able to mark attendance for forty students in under sixty seconds. A school owner should be able to see how much fee has been collected this month in two clicks. A parent should receive an SMS the moment their child is marked absent, without anyone at the school having to remember to send it. These things happen because the system is designed around real school workflows, not around software conventions.

Four types of people use the platform, and each one has a portal built specifically for their role. The Super Admin is the person who operates the platform itself — they manage which schools are on the system, handle billing, and have a god-view of all infrastructure health. The School Admin is the principal or owner of a specific school — they configure their institution, manage their staff, oversee finances, and generate reports. The Teacher is the classroom professional — they mark attendance, enter grades, create homework, and communicate with parents. The Student and Parent share a portal where they can see academic results, view the timetable, pay fees, submit homework, and receive all communications from the school.

From a technology standpoint, the platform runs entirely on free cloud infrastructure tiers. Vercel handles the frontend and API routing. Railway and Render host the application servers. Supabase provides the primary database. Upstash Redis handles caching and the message queue. Cloudinary stores all files. The total monthly cost to operate the platform for up to fifty schools is zero rupees. This is not a compromise — it is a deliberate architectural achievement that makes the platform commercially viable from day one without requiring infrastructure investment.
2. Super Admin — Platform Operations
The Super Admin is the operator of the SaaS platform. This role exists above all schools and manages the platform itself rather than any single institution.
School onboarding — When a new school signs up for the platform, the Super Admin uses an onboarding wizard to create that school's isolated tenant environment. They enter the school's name, upload its logo, set the academic calendar, and generate the initial credentials for the school's own administrator. This entire process takes under two minutes. Once complete, the new school is fully operational and its data is completely isolated from every other school on the platform — they share no tables, no files, and no application state.

Billing and subscription management — The Super Admin can see every school's subscription tier, payment history, and current status. If a school's subscription lapses, the Super Admin can suspend their access in one action, which immediately prevents all logins for that school while preserving all their data. Reactivation is equally instant. No data is ever deleted on suspension — only access is gated.

System health dashboard — The Super Admin has a real-time view of the platform's infrastructure: which application shards are healthy, how many database connections are active, the current message queue depth, error rates across all three shards, and P95 API response times. This dashboard is powered by Prometheus metrics scraped from each shard and visualised in Grafana Cloud. The Super Admin sees a problem before any school reports it.

Global announcements — When the platform needs to communicate with all school administrators simultaneously — for a scheduled maintenance window, a new feature launch, or a policy change — the Super Admin can send a platform-wide announcement that appears in every school admin's notification centre simultaneously.

Immutable audit trail — Every action performed by any Super Admin — school creation, suspension, configuration changes, support access — is recorded in an append-only audit log with the actor's identity, timestamp, and the exact change made. This log cannot be modified or deleted, providing a permanent compliance record of all platform-level operations.
3. School Admin — Institutional Management
The School Admin is the principal, owner, or management committee representative of a specific school. Their portal is the command centre for running the institution day to day.
3.1 School Configuration
School profile — The School Admin maintains the institution's master profile: its official name, logo, address, phone number, affiliation board, principal's name, and UDISE code. These details appear automatically on all generated documents — report cards, fee receipts, transfer certificates, and letters — so there is never a risk of stale school information on official output.

Academic calendar — The admin defines the academic year's start and end dates, term boundaries, public holidays, and school working days. This calendar is the backbone of every time-based feature in the system. Attendance is only tracked on working days. Fee due dates respect holidays. Exam schedules are validated against the calendar to prevent scheduling exams on non-working days.

Class and section management — The admin creates classes (Grade 1 through Grade 12, or equivalent) and sections within each class (Section A, B, C). Each class-section combination is an independent unit with its own teacher assignment, timetable, student roster, and fee structure. Adding or reorganising classes at the start of a new academic year takes minutes.
3.2 Staff Management
Staff hiring and profiles — Every teacher and non-teaching staff member has a comprehensive profile in the system: full name, employee ID, date of joining, designation, department, qualification, experience, PAN number, Aadhaar number, and bank account details for payroll. The profile also stores the staff member's photo and all employment-related documents uploaded as secure files.

Subject and class assignment — The admin assigns each teacher to the subjects they teach and the class sections they are responsible for. A single teacher can be assigned to multiple subjects across multiple class sections. When a teacher is assigned to a class, they immediately gain the ability to mark attendance, enter grades, and create homework for that class — no further configuration needed.

Offboarding workflow — When a staff member leaves the school, the admin can offboard them in a single action. This removes their login access, reassigns their classes to other teachers if needed, and preserves all historical records they created — attendance they marked, grades they entered, homework they set — since these records belong to the students, not the teacher.
3.3 Student Admissions
Admissions workflow — The admissions process is entirely digital. A prospective student's guardian fills an online application form capturing the child's personal details, family information, previous school history, and any special requirements. The admin receives the application, reviews it, requests any missing documents through the platform's messaging system, and either approves or rejects it. On approval, the system automatically assigns a class, generates a student ID, creates login credentials for the student and parent portal, and sends a welcome SMS to the registered parent's mobile number — all in a single click.

Document management — Every student's admission requires a set of documents: birth certificate, previous school transfer certificate, caste certificate if applicable, medical fitness certificate, and passport-size photographs. All of these are uploaded to the platform as part of the admissions process and stored permanently in Cloudinary's cloud storage. The admin can access, download, or print any document from any student's profile at any time, from any device.

Student ID generation — Each enrolled student receives a system-generated student identity card as a downloadable PDF. The card includes the student's photograph, name, date of birth, class and section, roll number, school name, academic year, and a QR code linking to their basic profile. Schools can print these in bulk at the start of each academic year.

Bulk import — For schools transitioning from another system or from paper records, the platform supports bulk import of student data from a standardised Excel template. An admin downloads the template, fills it with existing student records, and uploads it. The system validates each row, reports errors clearly, and on confirmation creates all student profiles simultaneously. A school with two thousand existing students can be fully migrated in under an hour.
3.4 Fee Structure and Finance Setup
Fee category definition — The admin defines all the fee categories that their school charges: tuition fee, transport fee, library fee, laboratory fee, sports fee, examination fee, and any school-specific categories. Each category has a name, description, and the amount applicable per class. Because different classes often have different fee amounts, the structure is defined per class rather than per school.

Payment schedule configuration — Once fee categories and amounts are defined, the admin sets the payment schedule: when each fee is due. The platform supports term-wise schedules (three payments per year), monthly schedules, quarterly schedules, or any custom schedule the school uses. Once configured, the system automatically generates individual fee records for every student enrolled in that class on the scheduled dates, without the admin having to do anything further.

Late fee rules — The admin can configure automatic late fee charges that apply when a student's payment passes its due date. The late fee can be a flat amount or a percentage of the outstanding balance, and it can be set to apply daily or on a one-time basis after the due date. The system calculates and applies late fees automatically — the admin sets the rule once and never has to think about it again.

Concession management — Schools frequently need to waive or reduce fees for specific students — children of staff members, scholarship recipients, siblings of existing students, or students from economically weaker sections. The admin can apply a named concession to any student's fee record, specifying the category, the discount amount or percentage, and a reason. All concessions are recorded with the admin's identity, preventing unauthorised waivers and providing a clear audit trail for any inspection.
3.5 Reports and Analytics
Financial reporting — The admin can generate a daily fee collection report showing every payment received on a given day, who collected it, which student paid it, and by what method. A monthly revenue summary shows total collection per fee category, total outstanding, and the collection efficiency percentage for the month. A term-end financial report consolidates all of this across the academic term and is the primary document used for school board presentations and audits.

Academic performance reports — At the end of each examination cycle, the admin can generate class-wise and school-wide academic performance reports. These show class average marks per subject, the distribution of grades across the class, subject-wise toppers, and the overall pass percentage. These reports can be downloaded as PDFs for circulation to the school board or as Excel files for further analysis.

Attendance reports — The admin has access to aggregated attendance data across the entire school. They can view class-wise attendance percentages for any date range, identify the classes with the highest absenteeism, and drill down to student-level attendance records. A monthly attendance summary is auto-generated on the last working day of each month and stored in the admin's report archive.
4. Attendance Management
Attendance is the most frequently used feature of the platform. Every teacher uses it every day. The design principle is that marking attendance should never take more than sixty seconds for a full class.
Daily roll call — When a teacher opens their attendance interface for a class on a given day, they see a pre-populated list of every student in that class section, arranged by roll number. All students default to present. The teacher only taps on the students who are absent, late, or have excused absences. For a class of forty students with typical absenteeism of two or three, the entire process takes under thirty seconds. The system records the date, class, teacher identity, and each student's status with a precise timestamp.

Bulk marking — On days when an entire class is absent — a school trip, a natural disaster, a declared holiday that was not in the original calendar — the teacher can mark all students absent in a single action and then selectively mark back those who are present. This inverted workflow is significantly faster than marking each absent student individually when most of the class is out.

Automatic parent SMS on absence — The moment a teacher marks a student absent and submits the attendance, the platform queues an SMS to the student's registered parent. This SMS is sent within thirty seconds of the teacher's submission and reads naturally: it includes the child's name, the class, the date, and the school's contact number in case the parent wants to call. The SMS is sent through the school's own IoT-based SMS gateway — a hardware device using an ESP32 microcontroller and SIM800L GSM module — which means there is no per-SMS cost to the school. Forty absent students generate forty SMS messages automatically, with no human intervention after the teacher submits attendance.

Attendance analytics — Every teacher and admin can view attendance analytics for any class over any date range. The analytics show the overall attendance percentage for the class, the trend over time as a chart, the days with the highest absenteeism, and a ranked list of students by attendance percentage. This makes it immediately visible which students are at risk of falling below the minimum attendance threshold required by the school or examination board.

Chronic absence detection — The platform automatically monitors every student's rolling thirty-day attendance record. When a student is marked absent for the third time within any thirty-day window, the system immediately creates an alert visible to both the class teacher and the school admin, and sends an internal notification prompting follow-up. This detection happens in the background via the worker service — the teacher does not have to check anything manually. A chronically absent student is surfaced automatically.

Minimum attendance warnings — Schools typically have a rule that students must maintain a minimum attendance percentage — often seventy-five percent — to be eligible for examinations. The platform tracks each student's running attendance percentage against this threshold and displays a warning flag on the student's profile and in teacher and admin views when the student drops below the configured minimum. Parents also receive a notification when their child crosses this threshold, giving time for intervention before the academic consequences take effect.

Historical attendance records — Every attendance record is permanently stored and queryable. A teacher can look up what the attendance was for their class on any day in the past academic year. An admin can pull the attendance record for a specific student across their entire enrollment history. These records feed directly into report cards, transfer certificates, and scholarship applications.
5. Timetable Management
The timetable module manages the weekly teaching schedule for the entire school, from initial creation through daily exceptions, substitute assignments, and exam schedules.
Visual timetable builder — The admin builds the school's weekly timetable through a visual interface that shows the week as a grid of days and periods. Each cell in the grid represents one period for one class section. The admin assigns a subject and teacher to each cell by selecting from dropdown menus of available subjects and teachers. The system performs real-time conflict detection: if the admin tries to assign a teacher to two different class sections at the same period, the system immediately highlights the conflict and prevents the save. The same applies to classroom conflicts in schools where rooms are shared.

Teacher timetable view — Every teacher sees their own personal timetable — a week-view of exactly which class they are teaching, in which subject, and in which room, for each period of each day. This view updates immediately whenever the admin makes a change to the master timetable. Teachers never need to check a notice board or ask anyone about their schedule.

Student timetable view — Students and parents see the class timetable for the student's enrolled class section. The view is clean and readable on a mobile screen. It shows the subject name and teacher's name for every period. When the admin makes any change — a period swap, a teacher reassignment, a room change — the student's timetable view reflects the update immediately.

Substitute teacher management — When a teacher is absent or on approved leave, the admin needs to assign a substitute for their classes that day. The admin opens the substitute management screen, selects the absent teacher, sees all their classes for the day highlighted, and assigns available teachers as substitutes for each affected period. Both the substitute teacher and all affected students receive an instant in-app notification informing them of the change. The substitute sees the affected class appear in their timetable for that day.

Period modification log — Every change to the timetable — a period swap, a substitute assignment, a permanent teacher reassignment — is recorded in a modification log with the admin's identity and a timestamp. This log serves as the official record of who authorised each schedule change, which is important for accountability in schools where teachers raise concerns about timetable fairness.

Exam timetable — The platform maintains a separate exam schedule that overlays on top of the regular timetable. The admin creates the exam schedule by specifying the date, time, subject, class section, exam hall, and invigilator for each examination session. The system validates that no student is scheduled for two exams simultaneously and that each invigilator is assigned to only one hall per session. Students see the exam timetable alongside their regular timetable, and the system sends reminder notifications two days and one day before each examination.

Hall ticket generation — For each student scheduled to sit an examination, the platform generates a printable hall ticket. The hall ticket shows the student's name, photograph, roll number, class, the examination centre, the exam hall, their assigned seat number, and the complete exam schedule for all their subjects. Hall tickets are generated as PDFs and can be printed individually or in bulk for the entire class.
6. Examinations and Grade Management
The examination module handles the complete lifecycle of assessments — from creating the exam, through entering marks, computing results, and generating report cards.
Examination configuration — The admin creates each examination in the system by specifying its name — Mid-Term Assessment, Annual Examination, Unit Test — along with the date range, the classes it covers, and the maximum marks for each subject in each class. The system supports both internal school examinations and board examinations. For board examinations, the system handles theory and practical marks separately if required.

Grading scheme configuration — The admin defines the school's grading scheme once. This might be a simple pass/fail threshold, a percentage-based letter grade system (A for ninety percent and above, B for seventy-five to ninety, and so on), or a grade point system as required by the affiliated board. Once configured, this scheme is applied automatically to every result in every examination — the admin never has to manually compute grades.

Mark entry by subject teachers — For each examination, the subject teacher enters marks for every student in their assigned classes. The mark entry interface is a simple grid: student names down the left, and each assessable component across the top. As the teacher enters marks, the system immediately validates them against the configured maximum — a teacher cannot accidentally enter a mark higher than the maximum or a negative value. Marks are saved progressively, so if a teacher is interrupted mid-entry, they do not lose work.

Automatic result computation — Once all subject marks are entered for an examination, the admin triggers result computation. The system calculates total marks, percentage, grade, grade point average, and pass/fail status for every student. It also computes class rank, subject-wise toppers, and overall school rank. All of this happens instantly — a school with five hundred students across ten classes gets fully computed results in under two seconds.

Performance analytics — The admin and class teacher can view performance analytics for each examination. These include the class average score per subject, the score distribution across the class shown as a histogram, the percentage of students who passed in each subject, and a comparison of performance against the previous examination. These analytics immediately surface subjects where the class performed poorly, prompting academic intervention before the next term.

Report card generation — The culmination of the examination module is the report card. For each student in each examination cycle, the platform generates a professionally formatted PDF report card. It carries the school's letterhead and logo, the student's photograph, name, class, and roll number, the marks obtained in every subject, the computed grade for each subject, the overall percentage and grade, the student's attendance percentage for the term, the class teacher's written remarks, and a signature block for the principal. Report cards are generated in bulk for the entire class in one action and are available for download by both the school and the student's parent portal.

Progress tracking across terms — The platform preserves examination results across academic terms and years. A student's performance history is always visible as a chronological record. Teachers and parents can see whether a student who struggled in the first term improved in the second, or whether a previously strong student is showing decline. This longitudinal view is available for every student throughout their enrollment.
7. Homework and Assignment Management
The homework module replaces the physical homework diary with a digital workflow that gives teachers more control and gives students and parents a clearer view of academic expectations.
Assignment creation — A teacher creates a new assignment by specifying the subject, the class section it is for, the assignment title, detailed instructions, and the due date. They can optionally attach files — a scanned worksheet, a PDF reading, a reference image — that students need to access to complete the work. The assignment is published to the student portal the moment the teacher saves it, and every student in that class section receives an in-app notification immediately.

Digital submission — Students submit their completed work directly through the platform. They can type their response as text for written assignments, or upload a file — a photograph of handwritten work, a PDF, a spreadsheet, or any other document — for assignments requiring physical work. The system records the submission timestamp for every student, making it straightforward to identify who submitted on time and who submitted late.

Late submission handling — Once the due date passes, students can still submit but their submission is automatically marked as late. The teacher sees late submissions highlighted distinctly from on-time submissions. The admin can configure school-wide late submission policies — whether late work is accepted at all, whether it receives reduced marks, or whether it is simply flagged but evaluated normally.

Teacher review and feedback — The teacher opens each submission from within the platform and reviews it in the browser — without downloading files. They enter a mark and written feedback for each student's work. Students receive an in-app notification the moment their submission is marked, and they can read the teacher's feedback directly in their portal. This closed loop — assign, submit, grade, feedback — all happens within the platform with no reliance on email or physical paper.

Assignment analytics — Teachers can see at a glance how many students have submitted for each assignment, how many have not submitted, and the distribution of marks across the class. The admin can see submission rates across all classes and all subjects, identifying teachers who regularly get high submission rates as well as subjects where students consistently struggle to complete work.
8. Finance and Fee Management
The finance module is the economic engine of the school's operations. It handles everything from fee definition through collection, receipts, defaulter management, payroll, and financial reporting — with full ACID compliance and an immutable audit trail for every transaction.
Automated fee record generation — When the admin configures a fee structure with a payment schedule, the system does not wait for someone to manually create fee records for students. Instead, on each scheduled due date, the system automatically generates an individual fee record for every student in the relevant class, with the correct amount, the correct due date, and a unique receipt-ready record number. For a school with eight hundred students and four fee components due quarterly, the system generates over three thousand records automatically throughout the academic year without any manual action.

Online fee payment — Parents can pay their child's outstanding fees directly through the parent portal using the integrated payment gateway. The platform supports Razorpay and Stripe, allowing payment by credit card, debit card, UPI, and net banking. When a payment is completed, the system immediately marks the fee record as paid, generates a numbered receipt, and sends a payment confirmation to the parent's registered mobile number and email address. The school's ledger is updated in real time — there is no reconciliation step needed.

Offline payment recording — For parents who pay in cash or by cheque at the school counter, the admin or fee collection staff member records the payment manually. They select the student, see all outstanding fees, select the fee being paid, enter the amount, choose the payment mode (cash, cheque, demand draft, or bank transfer), and optionally enter a reference number. The system generates the receipt immediately, and the ledger updates in real time exactly as it would for an online payment.

Partial payment support — Schools sometimes accept partial payment from families in financial difficulty. The platform handles this correctly — a partial payment marks the fee record as partially paid, carries forward the outstanding balance, and continues to calculate late fees on the outstanding portion. The parent's portal shows exactly how much has been paid and how much remains for each fee component.

Receipt management — Every payment, whether online or offline, generates a uniquely numbered, printable receipt in PDF format. The receipt carries the school's letterhead and logo, the student's name and class, the fee category paid, the amount, the payment method, the date and time of payment, the receipt number, and the name of the collection staff member for offline payments. These receipts are permanently stored against the student's financial record and can be reprinted or emailed to the parent at any time.

Defaulter management — The finance dashboard shows a real-time defaulter list — every student who has an overdue fee payment. The list shows the student's name, class, the fee category overdue, the original due date, the amount outstanding, the number of days overdue, and the cumulative late fee accrued. The admin can filter this list by class, by overdue amount, or by days overdue, making it easy to prioritise follow-up. The entire list is exportable to Excel for use in parent meetings or board presentations.

Automated demand notices — Rather than asking the admin to manually follow up with every defaulter, the platform can send automated fee demand notices on a configured schedule. The admin sets up a rule — for example, send a reminder email and SMS on the day after the due date, then again seven days later, then again fourteen days later with an escalating tone. The system sends these notices automatically to the defaulter's registered contact details without any further admin action.

Payroll processing — The platform extends its financial capabilities to staff payroll. The admin defines each staff member's gross salary, the applicable deductions — professional tax, provident fund, and income tax deducted at source — and any monthly allowances. On payroll processing day, the admin clicks a single button and the system computes net salary for every staff member, applying all configured deductions and allowances. Salary slips are generated as PDFs for every employee, downloadable by the admin and optionally sent directly to the staff member's email.
9. Communication System
The communication system is what connects the school to every parent. It is not an afterthought — it is a core infrastructure layer that delivers time-sensitive information reliably, instantly, and at zero per-message cost.
IoT-based SMS gateway — The platform's most distinctive technical feature is its custom SMS gateway. Rather than using a cloud SMS service that charges per message — which becomes expensive at scale with hundreds of daily absence alerts — the platform uses a hardware device consisting of an ESP32 microcontroller and a SIM800L GSM module connected to a standard SIM card. This device sits at the school or at the system administrator's location and handles all outgoing SMS traffic. The per-SMS cost is effectively zero — only the cost of a basic SIM plan applies. This gateway handles bulk SMS for absence alerts, fee reminders, exam notifications, and any other automated message the platform generates.

Absence alerts — As described in the attendance section, the moment a teacher submits attendance with absent students, the system queues an SMS for each absent student's registered parent. The SMS is delivered within thirty seconds. It is personalised — it says the student's name, not a generic message — and it includes the school's contact number so the parent can call immediately if needed. This feature alone substantially reduces the administrative burden on front-office staff who would otherwise spend hours every morning calling parents.

Fee reminders — The communication system sends fee reminders automatically based on the schedule configured by the admin. A typical configuration might send a gentle reminder one week before the due date, a direct reminder on the due date itself, and escalating notices at one week and two weeks after the due date. Every reminder is personalised with the student's name, the specific fee category, the amount due, and the due date. Parents never have to be chased in person — the system handles it.

Exam notifications — When the exam timetable is finalised, the system automatically sends exam reminder notifications to all students and their parents. The first reminder goes out one week before the first exam, and a second reminder goes out the evening before each exam. The notification includes the exam name, the date, the time, the subject, and the venue. Parents who are not always watching the school app still get the information via SMS.

School announcements — The admin can compose an announcement and specify its audience — the entire school, a specific class, or a specific group of parents. The announcement is delivered simultaneously across three channels: as an in-app notification in the recipient's portal, as an SMS to their registered mobile number, and as an email to their registered address. This three-channel delivery ensures critical information reaches parents regardless of which channel they are most attentive to.

Parent-teacher messaging — The platform includes a secure, persistent messaging system between teachers and individual parents. A parent can send a message to their child's class teacher directly from the parent portal. The teacher receives it in their messaging inbox and can reply. All messages are stored permanently, creating a record of all parent-teacher communication that can be referenced if any dispute arises about what was communicated and when.

Notification centre — Every user of the platform — admin, teacher, student, and parent — has a notification centre in their portal. This is the single inbox for all communications directed at them: system alerts, teacher messages, admin announcements, homework notifications, exam reminders, and payment confirmations. Notifications are marked as read when opened, giving the user a clear view of what they have and have not yet seen. The notification centre is also accessible on mobile browsers without a native app.
10. Student Information System
The student information system is the master record for every student enrolled in the school. Every other module — attendance, grades, fees, homework, communication — reads from and writes to this central student record.
Comprehensive student profiles — Each student's profile in the platform is far more complete than a simple name-and-class record. It holds the student's full name, date of birth, gender, category (general, OBC, SC, ST), religion, nationality, mother tongue, blood group, Aadhaar number, current address, and permanent address. It includes a photograph and stores the admission date and the unique student ID assigned at enrollment.

Family information — The student's profile links to a family record containing details of the father, mother, and guardian separately. For each family member, the platform stores their full name, occupation, employer name, annual income, mobile number, email address, and WhatsApp number if different from the mobile. It also captures their Aadhaar number for government scholarship and eligibility purposes. Single-parent and guardian-care scenarios are fully supported — the system does not force entry of both parents.

Document repository — Every document submitted during the admissions process — and any document added later — is stored permanently against the student's profile. The admin and school admin can view, download, and re-upload any document at any time. Documents stored include the birth certificate, transfer certificate from the previous school, caste certificate, medical fitness certificate, income certificate for fee waiver purposes, and any government scholarship eligibility documents. Because documents are stored in Cloudinary's cloud storage, they are never lost even if the school's local computers are lost or damaged.

Academic history — The student's profile aggregates their complete academic history from the day of enrollment: every examination result, every attendance record, every homework completion record, and every teacher remark. This history is available to the admin and the student's parents through the parent portal, and it travels with the student across academic years — a single profile maintains continuity from Grade 1 through Grade 12.

Transfer certificate generation — When a student leaves the school — whether transferring to another institution or completing the final grade — the admin can generate an official Transfer Certificate from the student's profile. The TC is auto-populated with all the required fields from the student record: student name, date of birth, mother tongue, category, class of study, date of admission, date of leaving, the student's behaviour and conduct rating, attendance percentage, and the principal's certification. The TC is generated as a professionally formatted PDF with the school letterhead, and a copy is permanently stored in the student's profile.

Health records — The platform supports optional health information for each student: blood group, known allergies, chronic medical conditions, vaccination status, and emergency medical contact. This information is visible to the admin and relevant teachers, and it can be critical in emergency situations at school. The platform does not mandate this information, but provides the structured place to store it when a school requires it.

Alumni management — Students who complete their schooling at the institution are not deleted from the system — they are moved to an alumni status. Their complete academic record, photographs, and documents are preserved. The school admin can search alumni records, generate duplicate certificates for past students, and in future cycles use the alumni database for reunion events or reference verifications.
11. Staff Management
The staff management module maintains complete records for every teacher and non-teaching employee at the school, integrating with payroll, leave management, and timetable assignment.
Staff profiles and documentation — Every staff member has a profile that mirrors the depth of a student profile: employment details, personal information, qualifications, work history, contact details, emergency contacts, and bank account information for payroll. Employment-related documents are stored securely — the appointment letter, experience certificates from previous employers, degree and diploma certificates, and professional certifications. These documents are accessible to the admin at any time and are never lost regardless of changes in administrative staff who manage the physical files.

Department and designation structure — The admin organises staff into departments — Academic, Administration, Accounts, Library, Support Staff — and assigns designations within each department. A designation carries a configured level of system permissions. A teacher designation gives access to attendance marking, grade entry, and homework creation. An admin designation gives access to the full administrative portal. The designation-to-permission mapping is what controls what each staff member can and cannot do within the system, without requiring the admin to individually configure permissions for each person.

Leave management workflow — When a teacher needs a day off, they submit a leave application through the platform specifying the dates, the type of leave (casual, earned, medical, or special), and a reason. The admin receives the application in their leave management queue and approves or rejects it with an optional comment. On approval, the teacher's leave balance is reduced, the timetable module is notified of the teacher's absence on those dates, and the admin is prompted to assign substitute teachers for the affected classes. The entire workflow is paperless and creates a permanent record of all leave taken by every staff member — essential for payroll and compliance.

Leave balance tracking — Every staff member's leave entitlement — the number of casual leave days, earned leave days, and medical leave days they are entitled to in the academic year — is configured by the admin at the start of the year. As leave is approved, the balance reduces. The staff member can see their remaining leave balance in their own portal at any time, and the admin sees the leave balance for every staff member in the leave management screen.

Performance records — The admin can add private performance notes and appraisal records to any staff member's profile. These notes are visible only to users with admin-level access and are never visible to the staff member themselves. They serve as the institutional memory of staff performance — important when management changes and the new principal needs historical context about each teacher.
12. Library Management
The library module manages the school's book collection — catalogue, issue, return, fine collection, and availability tracking — replacing paper ledgers with a searchable digital system.
Book catalogue — Every book in the school library is catalogued in the system with its title, author or authors, ISBN, publisher, year of publication, edition, subject category, and the number of physical copies the school owns. Books can be searched by any of these fields in real time. The catalogue is the foundation of the entire library workflow — without accurate cataloguing, the issue and return tracking cannot function correctly.

Issue and return workflow — When a student or staff member wants to borrow a book, the librarian looks up the member by their ID, selects the book from the catalogue, and records the issue. The system immediately records the issue date, calculates the due date based on the configured lending period (typically fourteen days for students, thirty days for staff), and decrements the available copies count. When the book is returned, the librarian records the return, the available count is restored, and the system automatically calculates whether any fine is due.

Fine calculation and collection — When a book is returned after its due date, the system calculates the fine by multiplying the number of overdue days by the configured fine rate per day. The librarian confirms the fine amount on the screen and collects it. The fine is recorded against the student's financial record and can optionally be added to their outstanding fee balance for consolidated collection at the fee counter, rather than requiring separate cash handling at the library desk.

Availability tracking — At any moment, the librarian or any admin can see exactly how many copies of any book are available for issue and how many are currently with borrowers. For books with multiple copies, this prevents the common scenario where a librarian issues the same book twice and only discovers the error when a second borrower cannot find it on the shelf.

Library reports — The library module generates reports that give the admin useful insights into library usage: the most borrowed books in any period, students or staff with overdue returns, total fine collected in a month, and the books that have never been borrowed — useful for identifying additions to the collection that did not serve their intended purpose.
13. Transport Management
The transport module manages the school's bus fleet, routes, and student allocations — integrating with the finance module to automatically handle transport fees.
Route and stop definition — The admin defines each bus route the school operates, listing the stops in sequence with the expected arrival time at each stop. A route might have fifteen stops, each with a scheduled pickup time. Once the route is defined in the system, it never needs to be re-entered — it is the permanent record of that route's operation.

Vehicle and driver records — Each school vehicle is registered in the system with its registration number, vehicle type, seating capacity, driver's name, driver's mobile number, and the fitness certificate expiry date. The system flags upcoming fitness certificate renewals so the admin is never caught with an expired certificate during a transport inspection.

Student allocation to routes — When a student is allocated to school transport, the admin selects their route and their boarding stop from the configured route. The system records this allocation and immediately adds the applicable transport fee to the student's fee schedule. The transport fee amount is configured per route as a distance-based tier — students who board at a closer stop pay less than students who board at a stop further from school. This fee assignment is automatic and requires no separate financial entry.

Driver route view — The driver of each bus has access to a simplified view of their daily route — the stops in order, the expected time at each stop, and the list of students boarding at each stop with their parent's emergency contact number. This view is optimised for use on a smartphone, with large text and minimal navigation.
14. Security and Data Protection
Security is not a feature added on top of the platform — it is built into every layer of the architecture. The platform is designed so that a data breach at one school is architecturally impossible to extend to another school, and so that an attack on the platform's infrastructure is stopped before it reaches any application code.
Multi-tenant data isolation — The most fundamental security property of the platform is that no user at School A can ever access data belonging to School B, regardless of what they do within the application. This isolation is enforced at the database layer through Supabase's Row-Level Security policies. Every query executed against the database automatically has a tenant ID filter applied by the database engine itself — it cannot be bypassed at the application layer. Even if an application bug existed that constructed a query without a tenant filter, the database would silently apply the filter anyway. This is the strongest possible form of tenant isolation.

JWT-based stateless authentication — Every user session is represented by a signed JSON Web Token rather than by a server-side session. This token is issued at login, stored in the user's browser, and sent with every subsequent request. The application server verifies the token's cryptographic signature on every request — a verification that takes under one millisecond and requires no database lookup. The token contains the user's identity, role, and tenant — so the server knows exactly who is making a request, what they are allowed to do, and which school they belong to, from the token alone.

Cloudflare Web Application Firewall — Before any request from the internet reaches the Vercel Edge or the application servers, it passes through Cloudflare's Web Application Firewall. The WAF has rule sets specifically designed to identify and block SQL injection attempts, cross-site scripting attacks, path traversal attempts, and hundreds of other known attack patterns. Requests that match these patterns are dropped at Cloudflare's network — they never reach the application. The WAF operates across Cloudflare's global network of three hundred edge locations, meaning attacks are stopped geographically close to their source rather than reaching the application's infrastructure.

DDoS protection — Cloudflare's automatic DDoS mitigation is always active. If an attacker attempts to overwhelm the platform with a high-volume request flood, Cloudflare's systems automatically detect the anomalous traffic pattern and begin absorbing the attack. The application servers experience no increase in traffic. This protection is in place for every school on the platform at no additional cost.

Per-tenant rate limiting — Even legitimate users are rate-limited to prevent any single school from consuming disproportionate server resources. Rate limits are enforced per tenant, per API route, using an atomic counter stored in Upstash Redis. A school that experiences an unusual traffic spike — perhaps due to a misconfigured script or a small-scale attack using compromised parent credentials — is rate-limited without affecting any other school's experience on the platform.

Complete audit trail — Every data modification in the platform — every student record created, every fee collected, every grade entered, every timetable change made — is written to an append-only audit log. The log records the actor's user ID and role, the tenant, the precise timestamp, the action performed, and the before and after state of the data. This audit log cannot be modified or deleted by any user of the platform, including the Super Admin. It is the definitive record of everything that happened in the system, available for regulatory inspection, dispute resolution, and forensic analysis.

HTTPS on all connections — All communication between users and the platform — whether through a browser, the mobile web app, or an API call — is encrypted using TLS. SSL certificates for all domains are automatically provisioned and renewed by Cloudflare and Vercel. There is no scenario in which an unencrypted connection to the platform is possible.

Data sovereignty and deletion — When a school offboards from the platform, their complete data is exported and handed to the school admin in a structured format before deletion. The deletion process removes all records, documents, and cached data associated with that tenant. The school's data does not linger on the platform after their relationship ends. This policy is configured in the data processing agreement available to all school admins.
15. How the Platform Works — Architecture in Plain Language
For school owners evaluating the platform and developers who will work on it, this section describes the technical architecture in plain language — not as a diagram, but as a coherent narrative of how every piece fits together.
The user's journey begins at Cloudflare — When a teacher opens the school ERP on their phone, their browser sends a request to the platform's domain. This request hits Cloudflare's global network — specifically, the Cloudflare data centre closest to where the teacher is physically located. If the teacher is in Mumbai, the request is handled by Cloudflare's Mumbai infrastructure, not by a server in the United States. Cloudflare validates the request against its WAF rules, checks that the domain is valid, and then routes the request onward. For requests that are purely for the frontend — the HTML, CSS, JavaScript, and images that make up the user interface — Cloudflare serves these from its own cache at that same edge location. This means the teacher's phone downloads the user interface in under fifty milliseconds regardless of where in India they are located.

API requests go through Vercel's Edge — When the teacher's app needs to actually do something — mark attendance, load student data, submit a form — it sends an API request. This request flows through Cloudflare to Vercel's edge network. At the Vercel edge, a piece of middleware code runs in under three milliseconds. This middleware is the platform's load balancer. It reads two things from the request: the API path (which tells it what operation is being requested) and a cookie containing the tenant identifier (which tells it which school the request is from). Using these two pieces of information, the middleware decides which of the three application server shards should handle the request. If the request is for a financial operation, it goes to Shard A on Railway. If it is for an attendance or timetable operation, it goes to Shard B on Railway. If it is for a student record or staff operation, it goes to Shard C on Render. The middleware performs a health check against the chosen shard before routing — if the shard is not responding, it automatically routes to one of the other shards instead.

The application shards do the actual work — The chosen shard receives the request and processes it. The shard is a stateless Express.js server running in a Docker container. Stateless means it holds no information about previous requests — each request arrives complete with everything needed to process it, principally the JWT token that identifies the user. The shard verifies the JWT signature, extracts the user's identity and role, applies any rate limiting, and then executes the business logic: marking the attendance, computing the fee, generating the report. Crucially, the shard does not store anything in its own memory beyond the duration of the request. When the response is sent and the connection closes, the shard has no memory that the request ever happened.

All data lives in shared external stores — Because the shards are stateless, all data must live somewhere external that every shard can reach. There are three such stores. Supabase is the primary PostgreSQL database — it receives all write operations, all inserts and updates, with full ACID compliance ensuring that no financial transaction is ever partially committed. Neon Postgres is a read-only replica of Supabase — all read operations, the queries that fetch data for display to users, are directed here so that read traffic and write traffic are handled by separate database servers and never compete with each other. Upstash Redis serves as both a cache and a message queue: frequently accessed data is cached in Redis for up to five minutes, meaning the database is not consulted for every page load of data that has not changed.

The message queue handles slow work in the background — Some operations triggered by a user request take time — sending forty SMS messages, computing attendance analytics for a five-hundred-student school, generating a PDF report. If the application shard had to do all of this work before responding to the teacher, the teacher would wait several seconds for simple actions. Instead, the shard does the minimum necessary work — saving the core data — and then publishes an event to the message queue in Upstash Redis. This publishing takes under one millisecond. The shard then responds to the teacher immediately. A separate worker service, also running on Railway, polls the message queue continuously. When it finds an event, it processes the slow work: sends the SMS messages, updates the analytics tables, generates the report. This background processing happens invisibly to the teacher, who received their confirmation the moment the core data was saved.

Observability makes everything visible — The platform is continuously monitored at two levels. Internally, each application shard exposes a metrics endpoint that a Prometheus server scrapes every fifteen seconds. These metrics track the number of requests per second, the error rate, the response time at the ninety-fifth percentile, the active database connection count, and the current message queue depth. All of these metrics are visualised in a Grafana dashboard that gives the Super Admin and developer a real-time view of the platform's health. Externally, a monitoring service called Better Stack pings each shard's health check endpoint from five locations around the world every thirty seconds. If any shard fails to respond correctly, Better Stack sends an alert within sixty seconds — faster than any school would notice and report the problem.

Everything is free — The final thing worth saying about the architecture is that every component described above runs on free-tier infrastructure. Vercel's free plan handles the frontend and API routing. Railway's free plan runs Shard A, Shard B, and the worker service. Render's free plan runs Shard C. Supabase's free plan provides five hundred megabytes of database storage and sixty concurrent connections. Neon's free plan provides three gigabytes of read-replica storage. Upstash's free plan provides ten thousand Redis commands per day. Cloudinary's free plan provides twenty-five gigabytes of file storage. Better Stack's free plan provides ten uptime monitors. Grafana Cloud's free plan provides unlimited dashboards with fourteen-day metric retention. The total monthly cost to operate the entire platform for up to fifty schools, serving thousands of daily users, is zero.
